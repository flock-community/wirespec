# Extract LSP server into `src/ide/lsp`

**Status:** Approved — ready for implementation plan
**Date:** 2026-05-24
**Scope:** PR 1 of a multi-PR effort. Subsequent PRs convert the extracted module to Kotlin Multiplatform; this PR is extraction only.

## Goal

Move the LSP server currently embedded in `src/ide/vscode/src/server.ts` into a new standalone module `src/ide/lsp/`, and have the VS Code extension consume it as an npm dependency. The runtime architecture (VS Code spawns a Node subprocess via the LSP client) is preserved; only the source-of-truth and packaging boundary changes.

## Non-goals (this PR)

- No Kotlin Multiplatform yet — the new module ships as plain TypeScript first.
- No npm publish — the module is `private: true` and is consumed locally via `file:` link.
- No IntelliJ plugin changes.
- No new LSP features, no behavior changes; `server.ts` moves verbatim.

## Motivation

- The VS Code extension currently mixes editor-specific client wiring (`extension.ts`) with editor-agnostic LSP server logic (`server.ts`). Editors other than VS Code (Neovim, Helix, Emacs, IntelliJ) cannot consume the latter today.
- A standalone LSP module is the prerequisite for the KMP rewrite, which will produce both a JS package (for VS Code and other LSP-capable editors) and a JVM artifact (consumable by the IntelliJ plugin).
- Doing the extraction as a separate, mechanical PR keeps risk low and isolates the KMP work to its own change set.

## Architecture

### Today

```
src/plugin/npm           Kotlin/Multiplatform → @flock/wirespec (npm)
  └─ file:← src/ide/vscode
              ├─ src/extension.ts   VS Code LanguageClient (extension host)
              └─ src/server.ts      LSP server (spawned subprocess via IPC)
                 ↳ esbuild → build/server.js → packaged into VSIX
```

### After PR 1

```
src/plugin/npm           unchanged → @flock/wirespec
  └─ file:← src/ide/lsp
              └─ src/server.ts      LSP server (moved verbatim)
                 ↳ esbuild → build/server.js  (this module's own build)

src/ide/lsp              new module, exports @flock/wirespec-lsp (private)
  └─ file:← src/ide/vscode
              └─ src/extension.ts   unchanged — still spawns build/server.js
                 ↳ esbuild bundles node_modules/@flock/wirespec-lsp/build/server.js
                   into the VSIX's build/server.js
```

The VS Code extension host still spawns a separate Node process via `TransportKind.ipc`; the produced VSIX is still self-contained.

                       main "build/server.js", bin "build/server.js",

## Build scripts

### `src/ide/lsp/package.json` scripts

```json
{
  "scripts": {
    "build": "npm run esbuild",
    "compile": "tsc",
    "esbuild": "esbuild src/server.ts --bundle --outfile=build/server.js --external:vscode --format=cjs --platform=node",
    "clean": "npm run clean:build && npm run clean:node_modules",
    "clean:build": "npx --yes rimraf build",
    "clean:node_modules": "npx --yes rimraf node_modules"
  }
}
```

### `src/ide/vscode/package.json` script changes

```diff
- "esbuild:server": "esbuild src/server.ts --bundle --outfile=build/server.js --external:vscode --format=cjs --platform=node",
+ "esbuild:server": "esbuild node_modules/@flock/wirespec-lsp/build/server.js --bundle --outfile=build/server.js --external:vscode --format=cjs --platform=node",
```

All other vscode scripts (`build`, `esbuild`, `esbuild:extension`, `vscode:package`, etc.) are unchanged. `extension.ts` is unchanged; it still references `context.asAbsolutePath(path.join("build", "server.js"))`, which still exists after the rebundled esbuild step.

## CI pipeline: `.github/workflows/build.yml`

The current `vscode` job is self-contained: it runs `./gradlew :src:plugin:npm:jsNodeProductionLibraryDistribution` (to materialize `productionLibrary/` that the file: link points at), then `npm i && npm run build` in `src/ide/vscode`. We extend the same job rather than introducing a new one — keeping with the existing one-job-per-deliverable style.

The new step is inserted between the Gradle library build and the vscode npm build:

```yaml
- name: Build LSP server
  working-directory: src/ide/lsp
  run: |
    npm i
    npm run build
```

So the `vscode` job's step order becomes: Gradle npm-library → lsp npm build → vscode npm build. Each step's outputs are visible to the next via the workspace.

The same step is added to `release-vscode` in the equivalent position (between the existing `jsNodeProductionLibraryDistribution` Gradle step and the `npm run vscode:publish` step).

A standalone `lsp` CI job (for parallelism and independent visibility) is intentionally deferred to PR 2, where the KMP/Gradle transition makes the natural unit a `./gradlew :src:ide:lsp:build` invocation.

## Testing

No automated LSP tests exist today; this PR does not add any (the KMP follow-up will, where the test surface stabilizes). Manual verification:

1. From a clean checkout: build `src/plugin/npm` (Gradle), `src/ide/lsp` (npm), then `src/ide/vscode` (npm). Each step succeeds.
2. Install the produced VSIX (`src/ide/vscode/wirespec.vsix`) in a development VS Code instance.
3. Open a `.ws` file:
   - Diagnostic regression check: introduce a syntax error; the error appears as a red squiggle with the same message as on master before the PR.
   - Semantic highlighting regression check: types and variables are colored as before.
   - Go-to-definition regression check: cmd-click on a custom type name jumps between its declaration and its usages.
4. Confirm CI runs `npm` → `lsp` → `vscode` in order.

- Add `:src:ide:lsp` to `settings.gradle.kts`.

- `jvmMain`: stdio over `System.in` / `System.out`. Produces a runnable jar.
- Delete `src/ide/lsp/src/server.ts` once parity is verified.

## Risks and mitigations

- **Risk:** `vsce package --no-dependencies` skips bundling `node_modules`. The bundled `build/server.js` already inlines all reachable code via esbuild, so this works today and continues to work after the move; the VSIX remains self-contained. Verified manually by inspecting the produced VSIX in the test plan.
- **Risk:** CI ordering — vscode building before lsp would fail at the new esbuild target path. Mitigated by the explicit `needs: lsp` declaration in the vscode CI job.
- **Risk:** `node_modules/@flock/wirespec-lsp/build/server.js` path is sensitive to npm install behavior with `file:` deps. npm copies file: deps into node_modules (not symlinks, in current npm versions), so the build path is stable.

## Out of scope

- Switching VS Code from IPC to stdio transport.
- Publishing `@flock/wirespec-lsp` to the npm registry.
- Adding LSP test infrastructure.
- Renaming or restructuring `src/ide/intellij-plugin`.
- Adding hover, completion, formatting, or any other LSP feature not present today.
